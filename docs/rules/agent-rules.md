# AI Agent Rules

## Agent 定位

本项目中的 Agent 指基于 LLM 的任务执行单元，包括聊天助手、RAG 问答、工具调用 Agent、多 Agent 协作和后续自动化工作流。

## 基本边界

- Agent 必须有明确目标、输入、输出、工具权限和失败兜底。
- Agent 不得默认拥有数据库写入、文件删除、外部请求、代码执行权限。
- 工具调用必须白名单化，参数必须校验，结果必须可审计。
- 所有高风险动作需要人类确认：删除数据、发送外部消息、执行付款、修改权限、发布部署。
- Agent 回答必须能区分事实、推测和建议。

## Prompt 规范

系统提示词必须包含：

- 角色和职责。
- 可用工具及限制。
- 输出格式。
- 安全边界。
- 出错时的处理方式。

禁止：

- 在提示词中硬编码密钥。
- 让模型绕过鉴权或数据隔离。
- 把用户隐私、长文档全文无限制塞入上下文。
- 要求模型“无论如何都执行”。

## RAG 规范

- 文档入库前必须记录来源、用户、文件名、类型、hash、chunk 数量。
- chunk 切分必须保留段落语义，避免固定长度硬切破坏上下文。
- 检索结果必须带 doc_id、chunk_id、score 和来源信息。
- 回答中需要能引用来源；没有可靠来源时说明“不确定”。
- RAG 结果不能替代权限校验，检索前必须按 user_id 或 tenant_id 过滤。

## Tool Calling 规范

每个工具必须定义：

- 名称：动词 + 领域，例如 `searchKnowledgeDocs`。
- 描述：何时使用，何时禁止。
- 参数 schema：类型、必填、枚举、长度限制。
- 返回 schema：成功、失败、空结果。
- 安全级别：只读、写入、外部副作用、高风险。

工具执行规则：

- 先校验参数，再执行业务。
- 工具失败要返回可解释错误，不抛给模型原始堆栈。
- 多工具链路需要 traceId。
- 写操作必须记录审计日志。

## 多 Agent 规范

- Supervisor 只负责拆解、分派、汇总和质量检查。
- Worker Agent 只处理单一职责。
- Agent 间传递结构化结果，避免长篇自然语言互相污染上下文。
- 循环协作必须设置最大轮次和退出条件。
- 任何 Agent 不得把另一个 Agent 的输出当作事实，必须校验关键结论。

## 观测与评估

- 记录 requestId、userId、conversationId、model、latency、token 估算、工具调用次数。
- 对失败原因分类：鉴权失败、检索为空、模型超时、工具失败、格式错误。
- 新 Agent 上线前至少准备 10 条回归用例。
- 评估维度：正确性、引用质量、拒答边界、延迟、成本、安全性。
